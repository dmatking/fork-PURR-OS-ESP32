# PURR OS — Driver & Module Architecture

> **Status:** Planned — refactor in place from v0.12.0 onward  
> **Kernel baseline:** KITT v0.9.0  
> **Last updated:** 2026-06-12

---

## Overview

PURR OS is moving toward a fully modular, plug-and-play architecture where the kernel, UI, drivers, and apps are independently versioned, independently built, and independently swappable. No component should have compile-time knowledge of another component's internals. Everything communicates through defined interface contracts.

The goal: adding a new device means writing a manifest file and dropping in a driver. Nothing else.

## Dependency Flow

```
apps
  ↓ talks to
ui
  ↓ talks to
kernel/interfaces   ←── kernel owns and versions these
  ↑ implemented by
drivers
  ↑ selected and wired by
devices + purrstrap
```

**Rules:**
- Nothing talks upward through layers
- Nothing skips a layer
- The kernel depends on nothing above driver level
- Drivers implement interfaces, they do not call into the kernel
- The UI calls the display interface, never a driver function directly

---

## Interface Design

Interfaces are **C function pointer structs**. One header per capability category. These headers are the API — they version independently from everything else and never break backwards compatibility within a major version.

### Example — `iface_display.h`

```c
typedef struct {
    const char   *name;
    esp_err_t   (*init)(const display_config_t *cfg);
    esp_err_t   (*push_pixels)(int x, int y, int w, int h, const uint16_t *data);
    esp_err_t   (*fill_rect)(int x, int y, int w, int h, uint16_t color);
    esp_err_t   (*set_brightness)(uint8_t level);
    void        (*get_info)(display_info_t *out);
    esp_err_t   (*deinit)(void);
} iface_display_t;
```

### Example — `iface_touch.h`

```c
typedef struct {
    const char  *name;
    esp_err_t  (*init)(const touch_config_t *cfg);
    bool       (*read_point)(uint16_t *x, uint16_t *y);
    bool       (*is_pressed)(void);
    esp_err_t  (*deinit)(void);
} iface_touch_t;
```

Each driver fills out the relevant struct and registers it with the kernel at boot. The kernel stores the pointer. All calls after boot go through that pointer — the kernel never calls a driver function directly.

## The Glue Layer

The glue layer wires a specific driver into the interface struct. It is **generated by purrstrap** from the device manifest — nobody writes it by hand.

Given a device manifest declaring `display = axs15231b`, purrstrap generates:

```c
// GENERATED — do not edit
// Generated by purrstrap for device: jc3248w535

#include "axs15231b/display_axs15231b.h"
#include "interfaces/iface_display.h"

static iface_display_t s_display = {
    .name           = "axs15231b",
    .init           = display_axs15231b_init,
    .push_pixels    = display_axs15231b_push_pixels,
    .fill_rect      = display_axs15231b_fill_rect,
    .set_brightness = display_axs15231b_set_brightness,
    .get_info       = display_axs15231b_get_info,
    .deinit         = display_axs15231b_deinit,
};

iface_display_t *glue_get_display(void) { return &s_display; }
```

The glue layer contains zero logic. It is pure wiring. The kernel calls `glue_get_display()` at boot and stores the pointer for the lifetime of the session.

## Device Manifest Format — `.pcat`

Each device has a `device.pcat` file. This is the only file you need to write to add a new device. purrstrap reads it, resolves drivers, validates versions, generates glue, and builds.

```toml
# devices/jc3248w535/device.pcat

[device]
name        = "jc3248w535"
chip        = "esp32s3"
flash_mb    = 16
psram       = true

[drivers]
display     = "axs15231b"
touch       = "axs15231b"
radio       = ""
gps         = ""

[ui]
framework   = "miniwin"

[pins]
# pin definitions reference pins.h in the same folder
```

## Driver Manifest Format — `.pcat`

Each driver folder contains its own manifest.

```toml
# drivers/display/axs15231b/driver.pcat

name            = "axs15231b"
version         = "0.0.1"
type            = "display"
iface_version   = "1"
idf_min         = "5.3.0"
chip            = ["esp32s3"]
kernel_min      = "0.8.0"
kernel_max      = ""

required_calls  = [
    "iface_display_t.init",
    "iface_display_t.push_pixels",
    "iface_display_t.fill_rect",
    "iface_display_t.set_brightness",
    "iface_display_t.deinit",
    "sys_drv_register",
]
```

### Fields

| Field | Description |
|-------|-------------|
| `name` | Unique driver identifier |
| `version` | Driver version, semver |
| `type` | Category: display, touch, radio, input, gps |
| `iface_version` | Which revision of the interface header this implements |
| `idf_min` | Minimum ESP-IDF version required |
| `chip` | List of supported chips |
| `kernel_min` | Minimum KITT version. `0.9.0` is the baseline for all drivers going forward |
| `kernel_max` | Maximum kernel version. **Empty = no ceiling**, ignored by purrstrap until explicitly set |
| `required_calls` | List of kernel/interface API calls this driver depends on |

## Versioning & Compatibility

### kernel_min / kernel_max

Every driver declares the kernel version range it is known to work with.

- `kernel_min` is always set. Baseline for all current drivers is `0.8.0`.
- `kernel_max` starts empty and is **only set when a breaking kernel change makes a driver incompatible.** Setting it is a deliberate act — it signals "this driver has not been validated above this kernel version."

### Compatibility Mode

When a driver's `kernel_max` is set and the current kernel exceeds it, purrstrap does **not hard fail**. Instead it runs a soft validation:

1. Walk the driver's `required_calls` list
2. Check every call against the current kernel's exported API
3. **All calls present** → build succeeds, driver runs in **compat mode**
4. **Any call missing** → hard fail with a clear message identifying the missing call and which driver needs it

### Driver Manager Status Badges

At runtime the driver manager displays a status badge per registered driver:

| Badge | Meaning |
|-------|---------|
| `[OK]` | Within kernel_min/max, fully validated |
| `[COMPAT]` | Beyond kernel_max, but all required_calls validated. Driver is running but may need an update |
| `[FAIL]` | A required_call was not found. Driver did not load |

The `[COMPAT]` warning is **persistent** — it shows every time the driver manager is opened, not just at boot. It is informational, not blocking. The intent is: *you're okay right now, but update this driver before the next kernel bump.*

### Summary of compat check flow

```
purrstrap build
  → driver kernel_max set AND current kernel > kernel_max?
      NO  → build normally, badge [OK]
      YES → walk required_calls
              all present? → build, badge [COMPAT], warn in driver manager
              any missing? → hard fail, print missing call name
```

## purrstrap Role

In the new model purrstrap is the **integrator**, not just a build wrapper.

Given a target device, purrstrap:

1. Reads `devices/<target>/device.pcat`
2. Resolves which driver packages are needed per category
3. Validates `kernel_min`, `kernel_max`, `idf_min`, `chip` for every driver
4. Runs the `required_calls` check if kernel_max is exceeded
5. Generates the glue layer source files
6. Injects driver paths into `EXTRA_COMPONENT_DIRS`
7. Calls the IDF build

Adding a new device = writing `device.pcat`. No cmake editing. No touching CoreOS.

## drvstrap (Future)

A separate tool for building and packaging driver blobs.

- Takes driver source + manifest
- Builds against a target chip and IDF version
- Outputs a precompiled `.a` + manifest as a distributable package
- Packages are chip-specific: `axs15231b_esp32s3_idf5.3.a`
- Distributed via the repo or a future driver registry

This allows drivers to be precompiled and dropped in without needing the driver source at all — true plug and play at the binary level.

## Migration Strategy

Refactoring in place from the current structure. The current repo is the starting point.

**Phase 1 — Define interfaces**
Write `iface_display.h`, `iface_touch.h`, etc. Headers only, no code. This forces the contracts to be decided before anything is moved.

**Phase 2 — Strip the kernel**
Remove device-specific blocks from `CoreOS/main/CMakeLists.txt`. Move them toward manifest-driven injection. CoreOS becomes device-agnostic.

**Phase 3 — Port one device end-to-end**
JC3248W535 is the reference implementation. Full manifest, generated glue, interface-driven. When this builds and runs cleanly under the new model, the pattern is proven.

**Phase 4 — Port remaining devices**
One at a time, following the reference.

**Phase 5 — UI decoupling**
MiniWin calls `iface_display_t` and `iface_touch_t` only. All direct driver calls removed from lib_miniwin.

## Current State vs Target

| Area | Current (0.11) | Target (0.12+) |
|------|---------|--------|
| Device config | Hardcoded in `CoreOS/main/CMakeLists.txt` | `device.pcat` manifest |
| Driver selection | cmake if/elseif tree | purrstrap resolver |
| Glue layer | Handwritten `hal_lcd.cpp` per device | purrstrap-generated |
| UI/kernel coupling | MiniWin compiled into kernel build | Independent `.purr` module |
| Driver dependencies | Conditional PRIV_REQUIRES (fragile) | Self-contained driver component |
| Versioning | None | semver + kernel_min/max + required_catcalls |
| Interfaces | None | catcall_* structs (C function pointer contracts) |
| Module format | None | `.purr` precompiled binary with `purr_module_header_t` |
| App tiers | `.claw`/`.paw` overloaded for both Lua and native | `.meow` (Lua), `.paws` (userland native), `.claw` (kernel-access native) |
